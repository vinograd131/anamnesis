# Классификатор тренировочных ограничений по анамнезу

Определяет по свободному тексту жалобы/анамнеза одну из 8 групп тренировочных ограничений,
чтобы фитнес-приложение или тренер могли адаптировать программу и снизить риск травм.

## Бизнес-задача

**Проблема.** Фитнес-приложения и онлайн-тренеры составляют программы, не зная медицинских
ограничений клиента. Жалобу клиент пишет свободным текстом («болит поясница, отдаёт в ногу»),
а не кодом МКБ — отсюда риск травм и претензий.

**Заказчик (ЦА).** B2B: фитнес-приложения, агрегаторы тренеров, ЛФК-сервисы. Альтернативы у них
сейчас — ручные анкеты-чеклисты (не понимают свободный текст) и общие дисклеймеры.

**Бизнес-цели.** Снизить риск травм и возвратов из-за неподходящей нагрузки; автоматизировать
онбординг (мгновенный скрининг жалобы вместо ручного разбора анкеты).

**Инженерные задачи.** Классификатор группы ограничений по свободному тексту жалобы + REST-сервис
для интеграции в приложение/CRM тренера.

## Группы ограничений

Позвоночник · Суставы · Кардио+невро · Дыхательные · Эндокрин/метабол · ЖКТ/брюшные ·
Мочеполовые/таз · Кожа.

Группы получены переформулировкой кодов МКБ-10 под задачу (см. `src/mapping.py`).

## Данные

`data/*.jsonl` — train (4627), dev (837), test (804). Поля: `idx`, `symptoms`, `code`.
Коды без отношения к тренировкам (D50/H65/Z00/F41) исключаются. Главная метрика — **macro-F1**
(классы несбалансированы, дисбаланс ~7×).

## Модели и результаты

Финальное сравнение — на отложенном **test** (804 примера, оценивается один раз). dev — для подбора.

| Модель | dev macro-F1 | test macro-F1 | test accuracy |
|---|---|---|---|
| tf-idf + LogReg (лемматизация) | 0.854 | 0.868 | 0.873 |
| **RuBioRoBERTa (файнтюн, LoRA)** | 0.854 | 0.865 | 0.870 |
| CatBoost на fastText | 0.833 | 0.843 | 0.852 |
| fastText + LogReg | 0.821 | 0.832 | 0.843 |
| RuBioRoBERTa (frozen) + MLP | 0.815 | 0.820 | 0.836 |

На чистом test лемматизированный tf-idf baseline и дообученная RuBioRoBERTa идут **вровень**
(~0.87, разница в пределах шума одного прогона). Преимущество трансформера проявляется не в
среднем по test, а в **устойчивости к свободной речи**: на перефразированном test трансформер
держит 0.774 против 0.653 у baseline (см. раздел робастности), и точнее на бытовых формулировках
(MFT). Поэтому в прод выбрана RuBioRoBERTa — реальные жалобы пишут свободным языком, а не как в
обучающем наборе.

## Запуск

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt

python -m src.baseline --split dev
python -m src.fasttext_clf --split dev
python -m src.catboost_clf --split dev
pytest
```

Трансформер (RuBioRoBERTa) считается на GPU — см. `notebooks/02_transformer_colab.ipynb`
для Google Colab. Тяжёлые зависимости — в `requirements-transformer.txt`.

## Сервис и развёртывание

`src/serve.py` — REST API (FastAPI). Эндпоинты:
- `GET /ping`, `GET /health` — проверки живости/статуса;
- `GET /groups` — все группы и их ограничения;
- `POST /predict` — `{"text": "..."}` → `{group, restriction, confidence}`.

Прод-модель — RuBioRoBERTa (`MODEL=transformer`), лёгкий fallback на tf-idf (`MODEL=baseline`).
Запуск в Docker (CPU, без GPU):

```bash
docker build -t anamnes .
docker run -p 8000:8000 anamnes   # Swagger UI: http://localhost:8000/docs
```

### Нефункциональные требования

Сценарий — онбординг одного клиента (одиночный запрос, не пакетная обработка), поэтому критична
задержка одиночного `/predict`, а не пропускная способность.

- **Требование:** медиана `/predict` ≤ 1 с на одну жалобу (интерактивный отклик).
- **tf-idf baseline (CPU):** измерено — медиана ~3 мс, p95 ~4 мс (лемматизация кэшируется).
- **RuBioRoBERTa (CPU):** тяжелее (~0.4B параметров), задержка измеряется на целевом развёртывании;
  для онбординга (один запрос) укладывается в требование, для нагруженного сценария запасной
  вариант — лёгкий tf-idf-режим.
