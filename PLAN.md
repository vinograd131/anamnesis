# План проекта: классификатор фитнес-ограничений по анамнезу

## 1. Продуктовая идея (критерий 4 — продуктовое мышление)

**Проблема.** Фитнес-приложения и онлайн-тренеры составляют программы, не зная медицинских
ограничений клиента. Жалобы клиент пишет свободным текстом («болит поясница, отдаёт в ногу»),
а не кодом МКБ. Итог — риск травм и претензий.

**Решение (MVP).** Сервис, который по тексту жалобы/анамнеза определяет **группу тренировочных
ограничений** (8 групп) и отдаёт её приложению/тренеру, чтобы адаптировать программу.

**8 групп ограничений** (свой артефакт — переформулировали 105 кодов МКБ под продукт):

| Группа | Ограничение в тренировке |
|---|---|
| Позвоночник | без осевой нагрузки (становая, присед со штангой) |
| Суставы | без ударных, контроль амплитуды/осевой на сустав |
| Кардио+невро | без ВИИТ и натуживания, контроль пульса/АД, координация |
| Дыхательные | ограничение аэробной интенсивности, триггеры |
| Эндокрин/метабол | адаптация объёма и режима |
| ЖКТ/брюшные | ограничение нагрузки на пресс / внутрибрюшного давления |
| Мочеполовые/таз | ограничение нагрузки на таз, натуживания |
| Кожа | как правило без ограничений |

**ЦА:** B2B — фитнес-приложения, агрегаторы тренеров, ЛФК-сервисы. **Конкуренты:** ручные
анкеты-чеклисты (не понимают свободный текст), общие дисклеймеры. **Импакт:** снижение
травм/возвратов, автоматизация онбординга клиента.

## 2. Цель ML и метрика успеха (критерий 2 — Data Science)

- **Задача:** многоклассовая классификация текста (8 групп).
- **Главная метрика:** **macro-F1** (классы несбалансированы — accuracy завышает за счёт крупных групп).
  Доп.: accuracy, per-group F1, confusion-матрица.
- **Достигнутый baseline (tf-idf + LogReg, dev):** accuracy 0.855 / **macro-F1 0.841**.
- **Точка отсчёта (most-frequent class):** ~10.6%.
- **Линейка моделей (по нарастанию сложности):**
  1. **tf-idf + LogReg** — baseline, референс.
  2. **fastText** — субсловные n-граммы, устойчивость к опечаткам/словоформам (русский).
  3. **CatBoost поверх fastText-эмбеддингов** — нелинейная модель на доменных векторах.
  4. **Трансформер без файнтюна** — замороженные эмбеддинги RuBioRoBERTa (mean-pooling / `[CLS]`)
     + LogReg. Показывает «сырую» силу доменного предобучения.
  5. **Трансформер с файнтюном** — RuBioRoBERTa, дообучение под задачу (full / LoRA).
  - Основной трансформер — **RuBioRoBERTa** (`alexyalunin/RuBioRoBERTa`): RoBERTa, предобученная
    на русских биомедицинских текстах (SOTA на RuMedBench) — совпадает с доменом анамнезов.
    CPU-fallback — `cointegrated/rubert-tiny2`, если облако недоступно.
- **Гипотеза проекта (честная):** прошлые эксперименты показали, что простая модель сильна.
  Поэтому строим честную лесенку из 5 моделей и сравниваем на тесте; **в прод идёт модель,
  победившая по метрике**, а не «обязательно трансформер». Это и есть зрелое инженерное решение.

## 3. Данные

- `data/train_v1.jsonl` (4640), `dev_v1.jsonl` (839), `test_v1.jsonl` (~813 после фильтра).
  Поля: `idx`, `symptoms`, `code`. Сплиты готовы, утечек между ними нет (проверено).
- Препроцессинг: маппинг `code → group` (файл `code_group_mapping_v1.csv`), исключение DROP-кодов
  (D50/H65/Z00), нормализация текста (нижний регистр, чистка).
- **test трогаем только в самом конце** (один раз, для финального сравнения). HPO — на dev.

## 4. Структура репозитория (критерий 1 — инженерия)

```
fitness-anamnesis/
├── data/                      # train/dev/test jsonl (как есть)
├── src/
│   ├── data.py                # загрузка, маппинг в группы, фильтр DROP
│   ├── mapping.py             # словарь code->group (+CSV), один источник правды
│   ├── baseline.py            # tf-idf + LogReg
│   ├── fasttext_clf.py        # fastText (субсловные n-граммы)
│   ├── mlp_catboost.py        # MLP / CatBoost поверх tf-idf или эмбеддингов
│   ├── transformer_frozen.py  # RuBERT без файнтюна: эмбеддинги + линейная голова
│   ├── transformer_ft.py      # RuBERT с файнтюном (full / LoRA) + Trainer
│   ├── tune_optuna.py         # HPO через Optuna
│   ├── evaluate.py            # метрики, confusion, отчёт
│   └── serve.py               # FastAPI: POST /predict -> группа + ограничения
├── notebooks/
│   └── 01_eda.ipynb           # EDA: распределение, длины, утечки, кластеры
├── tests/                     # pytest: маппинг покрывает все коды, формат, инференс
├── reports/                   # графики (PNG), confusion, таблица сравнения
├── Dockerfile                 # образ для инференса
├── .github/workflows/ci.yml   # CI: линт + тесты
├── requirements.txt
├── README.md                  # продукт, результаты, как запустить
└── PLAN.md
```

Качество кода: `ruff` (линт) + `black` (формат), типизация ключевых функций, `pytest`.

## 5. Этапы (каждый = Plan → Do → Verify)

**Этап 0. Каркас.** Создать репо, `git init`, `.venv`, `requirements.txt`, структура папок.
*Verify:* `pip install` проходит, `pytest` (пустой) зелёный.

**Этап 1. Данные и маппинг.** Перенести `code_group_mapping_v1.csv` в `src/mapping.py`,
функции загрузки и группировки, фильтр DROP.
*Verify:* тест «все коды из data покрыты маппингом», размеры train/dev/test совпадают с ожидаемыми.

**Этап 2. EDA (ноутбук).** Распределение групп, длины текстов, дисбаланс, проверка утечек,
визуализация лексической близости групп. Графики → `reports/`.
*Verify:* перечитать выводы, графики строятся без ошибок.

**Этап 3. Baseline — tf-idf + LogReg.** tf-idf (1–2 граммы) + LogReg(class_weight=balanced).
Метрики на dev, confusion-матрица. Зафиксировать как референс.
*Verify:* macro-F1 ≈ 0.84 воспроизводится, артефакты сохранены.

**Этап 4. fastText.** Субсловные n-граммы (устойчивость к словоформам/опечаткам), supervised-режим.
Метрика на dev, сравнение с baseline.
*Verify:* модель учится, macro-F1 на dev считается, артефакт сохранён.

**Этап 5. CatBoost на fastText.** CatBoost (`auto_class_weights=Balanced`) поверх усреднённых
fastText-эмбеддингов (переиспользуем `models/fasttext.model`). Метрика на dev.
*Verify:* macro-F1 на dev считается, артефакты сохранены.

**Этап 6. Трансформер без файнтюна.** RuBioRoBERTa (`alexyalunin/RuBioRoBERTa`): замороженные
эмбеддинги (mean-pooling / `[CLS]`) + LogReg. Эмбеддинги извлечь один раз и закэшировать
(на CPU медленно, но разово). Метрика на dev.
*Verify:* эмбеддинги извлекаются и кэшируются, macro-F1 на dev считается.

**Этап 7. Трансформер с файнтюном.** Та же RuBioRoBERTa, дообучение под задачу (full или LoRA через
`peft`) через HF `Trainer`. **Считать на GPU (Colab/Kaggle)** — на CPU неподъёмно. HPO через
**Optuna** (`learning_rate`, epochs, batch_size, weight_decay, + ранг/alpha/dropout для LoRA),
цель — dev macro-F1, ~20–40 трайлов с pruning. Fallback при недоступности GPU — `rubert-tiny2` на CPU.
*Verify:* loss падает, Optuna-стади сохранён, лучший трайл воспроизводится.

**Этап 8. Честное сравнение.** Все 5 моделей — **на test, один раз**. Несколько seed'ов, разброс
метрик, таблица + confusion. Вывод: какая модель в прод и почему.
*Verify:* цифры сходятся между запусками, вывод обоснован.

**Этап 9. MVP-инференс.** `serve.py` (FastAPI): принимает текст → возвращает группу +
текстовое описание ограничений. Победившая модель.
*Verify:* `POST /predict` отвечает на 3–5 примерах корректно.

**Этап 10. Инженерия.** `Dockerfile` (инференс), CI (линт + тесты), README с результатами и
инструкцией запуска.
*Verify:* `docker build` собирается, CI зелёный, README воспроизводим.

**Этап 11. Презентация.** Слайды/отчёт под 5 критериев: проблема → данные → модели → результат →
продукт. График прогресса macro-F1 по 5 моделям (0.33 → ...).
*Verify:* прогон по чек-листу критериев, ответы на вероятные вопросы комиссии.
